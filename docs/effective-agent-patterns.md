> [!info] 核心观点
> 最成功的 Agent 实现并非依赖复杂框架，而是基于简单、可组合的模式。在增加复杂性之前，先用最简单的方案解决问题。

Anthropic 在过去一年与数十个团队合作构建 LLM Agent 后发现：成功的实现往往使用简单的、可组合的模式，而非复杂框架。本文总结他们从客户实践和内部开发中学到的经验。

---

## Agent 的定义

"Agent" 这个词有多种定义。有人指完全自主的系统，有人指按预设流程执行的程序。Anthropic 将所有这些变体统称为 **agentic systems**，但区分两种架构：

- **Workflow（工作流）**：LLM 和工具通过预定义的代码路径编排
- **Agent（代理）**：LLM 动态指挥自己的流程和工具使用，保持对任务完成方式的控制

理解这个区别很重要——**Workflow 追求可预测性，Agent 追求灵活性**。

---

## 何时使用（何时不使用）Agent

> [!warning] 先追求简单
> 用 LLM 构建应用时，推荐先找最简单的方案，只有在必要时才增加复杂性。很多情况下，优化单个 LLM 调用加上检索和示例就足够了。

Agent 系统通常用**延迟和成本换取更好的任务表现**。需要权衡这个代价是否值得：

- **Workflow 适用场景**：任务明确、需要可预测性和一致性
- **Agent 适用场景**：需要灵活性、需要模型做决策的大规模任务
- **简单方案优先**：对于多数应用，优化的单次 LLM 调用 + 检索 + 上下文示例就够了

---

## 框架的选择

市面上有很多框架简化 Agent 系统的实现：Claude Agent SDK、AWS Strands Agents SDK、Rivet（可视化工作流构建器）、Vellum 等。

这些框架简化了底层任务（调用 LLM、定义工具、链式调用），但也带来问题：

- 增加抽象层，使得底层 prompt 和 response 难以调试
- 容易引诱开发者在简单方案就够用时增加复杂性

**建议**：先用 LLM API 直接实现——很多模式只需几行代码。如果用框架，确保理解底层代码。**对底层行为的错误假设是客户错误的常见来源**。

## 构建模式详解

从基础构建块开始，逐步增加复杂性。

### 基础构建块：增强型 LLM

Agent 系统的基础是增强版 LLM——具备检索、工具、记忆等能力。当前模型可以主动使用这些能力：生成搜索查询、选择合适工具、决定保留什么信息。

![[图片存放/f58aedb9c99227ca6673e35d04112905_MD5.webp]]

**实现建议**：
- 根据具体用例定制这些能力
- 为 LLM 提供清晰、文档完善的接口

可以通过 [Model Context Protocol (MCP)](https://www.anthropic.com/news/model-context-protocol) 实现这些增强能力，它能用简单的客户端实现集成不断增长的第三方工具生态。

---

### Workflow 1：Prompt Chaining（提示链）

将任务分解为一系列步骤，每个 LLM 调用处理前一个的输出。可在任意中间步骤添加程序化检查（"gate"）确保流程正确。

![[图片存放/490235832c9cd47640a3e84c0fdffc59_MD5.webp]]

**适用场景**：任务可以清晰分解为固定子任务，愿意用延迟换取更高准确性。

**典型案例**：
- 生成营销文案 → 翻译成其他语言
- 写文档大纲 → 检查大纲是否符合标准 → 基于大纲写文档

---

### Workflow 2：Routing（路由）

对输入分类，导向专门的后续任务。这实现了关注点分离，可以构建更专业的 prompt。没有这个 workflow，针对一种输入优化可能损害其他输入的表现。

![[图片存放/ba5f2ff8f4c95476a57f48b6c67a928c_MD5.jpg]]

**适用场景**：复杂任务有明显类别需要分别处理，且分类可以准确完成。

**典型案例**：
- 客服查询路由：一般问题、退款请求、技术支持 → 不同流程、prompt、工具
- 成本优化路由：简单问题 → Haiku，复杂问题 → Sonnet

---

### Workflow 3：Parallelization（并行化）

LLM 同时处理任务，输出程序化聚合。两种变体：

- **分片（Sectioning）**：将任务分解为独立子任务并行执行
- **投票（Voting）**：同一任务多次执行，获得多样化输出

![[图片存放/1bfb3092e61bde28c61a0f61a4306f2c_MD5.webp]]

**适用场景**：子任务可并行加速，或需要多角度/多次尝试提高置信度。对于多考虑因素的复杂任务，让每个 LLM 调用专注一个方面效果更好。

**典型案例**：
- **分片**：一个模型处理用户查询，另一个检查不当内容（比同一个调用处理两者更好）
- **投票**：代码安全审查，多个不同 prompt 各自检查并标记问题

---

### Workflow 4：Orchestrator-Workers（编排器-工作者）

中央 LLM 动态分解任务、委托给工作者 LLM、综合结果。

![[图片存放/1728d05e0b6e9cb9cd23cc33020fb82c_MD5.webp]]

**与并行的关键区别**：子任务不是预定义的，由编排器根据输入动态决定。

**适用场景**：无法预测需要哪些子任务的复杂任务。

**典型案例**：
- 编程产品：对多个文件进行复杂修改
- 搜索任务：从多个来源收集分析信息

---

### Workflow 5：Evaluator-Optimizer（评估器-优化器）

一个 LLM 生成响应，另一个提供评估和反馈，形成循环。

![[图片存放/4e67fc6ac74bae4f2ab5d3ec62c1d8d6_MD5.webp]]

**适用场景**：有清晰评估标准，迭代改进能带来可衡量价值。两个迹象表明适合用此模式：
1. 人类反馈能明显改进 LLM 响应
2. LLM 能提供这样的反馈

这类似于人类写作者迭代打磨文档的过程。

**典型案例**：
- 文学翻译：翻译器可能遗漏细微差别，评估器提供批评
- 复杂搜索：多轮搜索分析，评估器决定是否继续搜索

---

### Agent（自主代理）

随着 LLM 在理解复杂输入、推理规划、可靠使用工具、从错误恢复等能力上成熟，Agent 开始在生产环境出现。

Agent 的工作流程：
1. 从用户命令或交互讨论开始
2. 任务明确后独立规划和执行
3. 可能返回向人类请求更多信息或判断
4. 执行中从环境获取"ground truth"（工具调用结果、代码执行结果）评估进展
5. 可在检查点或遇到阻碍时暂停等待人类反馈
6. 任务完成或达到停止条件（如最大迭代次数）时终止

![[图片存放/b37f65c63f05a63ef73b5cd237e729fd_MD5.webp]]

**实现本质**：Agent 往往只是 LLM 在循环中根据环境反馈使用工具。因此，**清晰设计工具集和文档至关重要**。

**适用场景**：开放性问题，无法预测所需步骤数，无法硬编码固定路径。需要信任 LLM 的决策能力。

> [!warning] 注意风险
> Agent 的自主性意味着更高成本和复合错误的可能。建议在沙盒环境广泛测试，并设置适当防护措施。

**典型案例**（Anthropic 内部实现）：
- 解决 SWE-bench 任务的编码 Agent
- "computer use" 参考实现

## 模式组合与定制

这些构建块不是规定性的，开发者可以塑形组合以适应不同用例。成功的关键是**衡量表现并迭代实现**——只有当复杂性确实改善结果时才添加它。

---

## 核心原则

构建 Agent 时遵循三个原则：

1. **保持简单**：Agent 设计力求简洁
2. **透明优先**：明确展示 Agent 的规划步骤
3. **精心设计 ACI**：通过完善的工具文档和测试，打造清晰的 Agent-Computer Interface

框架能帮你快速起步，但走向生产时不要犹豫减少抽象层，用基础组件构建。遵循这些原则，你能创建既强大又可靠、可维护、值得信赖的 Agent。

## 附录：实践案例

### 客户支持

自然融合聊天机器人界面与工具增强能力。适合 Agent 的原因：
- 支持交互自然遵循对话流程，同时需要访问外部信息和执行动作
- 工具可集成客户数据、订单历史、知识库文章
- 退款、更新工单等动作可程序化处理
- 成功可通过用户定义的解决方案清晰衡量

一些公司已采用按成功解决收费的定价模式，显示对 Agent 效果的信心。

### 编码 Agent

软件开发是 LLM 特性的潜力领域，从代码补全演进到自主解决问题。Agent 特别有效的原因：
- 代码解决方案可通过自动化测试验证
- Agent 可用测试结果作为反馈迭代解决方案
- 问题空间定义明确、结构化
- 输出质量可客观衡量

Anthropic 的实现中，Agent 已能仅根据 PR 描述解决 SWE-bench Verified 基准测试中的真实 GitHub issue。但自动化测试只验证功能，人工审查仍对确保方案符合系统需求至关重要。

## 附录：工具的 Prompt 工程

无论构建哪种 Agent 系统，工具都是重要组成部分。工具定义和规范应该与整体 prompt 一样受到重视。

**格式选择原则**：
- 给模型足够 token "思考"，避免把自己写入死角
- 格式接近模型在互联网上自然见到的
- 避免格式"开销"（如需要精确计算数千行代码、字符串转义）

**ACI 设计建议**（Agent-Computer Interface，类比 HCI）：
- **换位思考**：作为模型，根据描述和参数，使用这个工具是否显而易见？好的工具定义包括示例用法、边界情况、输入格式要求、与其他工具的清晰边界
- **优化命名和描述**：像给团队初级开发者写优秀 docstring 一样，尤其使用多个相似工具时
- **测试工具使用**：在 workbench 运行多组示例输入，观察模型犯错并迭代
- **Poka-yoke（防错设计）**：改变参数让犯错更难

**案例**：Anthropic 在构建 SWE-bench Agent 时，发现模型在 Agent 离开根目录后会在相对路径上犯错。修复方法是将工具改为始终要求绝对路径——之后模型使用无误。

---

## 关键要点

- **Workflow 追求可预测性，Agent 追求灵活性**——理解这个区别是选择模式的前提
- **先追求简单**：多数任务优化的单次 LLM 调用就够了，不要过度工程化
- **五个 Workflow 模式**：Prompt Chaining（顺序分解）、Routing（分类路由）、Parallelization（并行处理）、Orchestrator-Workers（动态分解）、Evaluator-Optimizer（迭代优化）
- **Agent 本质简单**：往往是 LLM 在循环中根据环境反馈使用工具，复杂在于工具设计
- **ACI 与 HCI 同等重要**：投入与人类界面同等的精力设计 Agent-Computer Interface
