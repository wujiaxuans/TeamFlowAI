# RAG 高级技术详解

> [!info] 核心定义
> **RAG（检索增强生成，Retrieval Augmented Generation）= 检索技术 + LLM 提示**。通过从自有垂域数据库检索相关信息，合并到提示词模板中，让大模型生成更准确的回答。这是解决大模型**知识局限性**、**幻觉问题**和**数据安全性**三大痛点的有效方案。

RAG 已经成为大模型应用最主流的实现方案：

![[图片存放/cf2fd9a2d222dd6876af74d803c4cd7c_MD5.jpg]]

完整的 RAG 流程分为两个阶段：**数据准备阶段**（数据提取→文本分割→向量化→数据入库）和**应用阶段**（用户提问→数据检索→注入 Prompt→LLM 生成答案）。

本文基于 LlamaIndex 的实现体系，系统讲解从原始 RAG 到高级 RAG 的核心技术，涵盖分块策略、索引优化、查询转换、智能体路由等八大高级技术。

## 为什么需要 RAG？

基础大模型在实际业务场景中存在三大局限：

| 问题 | 原因 | RAG 的解决方案 |
| --- | --- | --- |
| **知识的局限性** | 训练数据仅限于公开网络数据，无法覆盖实时性、非公开或私域数据 | 通过检索自有数据库，注入私有知识 |
| **幻觉问题** | 深度学习模型基于概率输出，容易"一本正经地胡说八道" | 检索到的真实信息作为上下文约束生成 |
| **数据安全性** | 企业不愿将私域数据上传第三方平台训练 | 数据保留在本地，仅传递检索结果给 LLM |

> [!example] 典型应用场景
> 客服机器人回答"石头扫地机器人 P10 的续航时间是多久？"——RAG 从产品文档中检索相关参数，注入 Prompt 后让 LLM 生成准确回答，而非依赖模型内部知识。

## 原始 RAG：基础流程

原始 RAG 的实现非常直接：

![[图片存放/d21c0549b5d0b2971bbfa658953102a8_MD5.png]]

1. **分块**：将文档拆分为固定大小的文本块
2. **向量化**：使用 Transformer Encoder（如 BERT、bge-large）将文本块嵌入为向量
3. **索引**：将所有向量存入索引（平面索引或向量数据库）
4. **检索**：用户查询向量化后，搜索余弦距离最近的 top-k 个向量
5. **生成**：将检索到的文本块作为上下文，注入 Prompt 让 LLM 回答

Prompt 注入示例：

![[图片存放/4b318d5e80d3646f8f6ca4868cfccab5_MD5.jpg]]

```python
# 简化版 RAG 提示词模板
def question_answering(context, query):
    prompt = f"""
    请基于以下上下文回答问题。如果上下文中没有相关信息，请说明。
    
    【上下文】
    {context}
    
    【问题】
    {query}
    
    【回答】
    """
    response = get_completion(prompt, model="gpt-3.5-turbo")
    return response
```

原始 RAG 的局限在于检索质量完全依赖于简单的向量相似度，无法处理复杂查询、多跳推理等场景。

### 1. 分块 (Chunking) & 向量化 (Vectorisation)

**分块策略**是 RAG 效果的第一道关卡。Transformer 模型有固定的输入序列长度限制，且长文本的向量表示会损失语义精度，因此需要将文档拆分为合适大小的块。

> [!info] 块大小的权衡
> - **小块**（如 256 tokens）：语义更聚焦，检索精度高，但可能丢失上下文
> - **大块**（如 2048 tokens）：保留更多上下文，但检索精度下降
> - **最佳实践**：根据 embedding 模型的限制（如 BERT 最多 512 tokens，OpenAI ada-002 最多 8191 tokens）和业务场景选择折中值

> [!tip] 想深入了解分块策略？
> 参见 [[data-loading-chunking|数据加载与文本分块]]，包含：
> - **8 种文档加载器对比**：PyMuPDF4LLM、Unstructured、LlamaParse、MinerU 等选型指南
> - **固定大小分块**：CharacterTextSplitter 段落感知实现
> - **递归字符分块**：RecursiveCharacterTextSplitter 分隔符层级策略
> - **语义分块**：SemanticChunker 通过嵌入距离检测主题边界
> - **结构感知分块**：MarkdownHeaderTextSplitter 保留标题元数据
> - **分块可视化工具**：ChunkViz 在线预览分块效果

**向量化**需要选择搜索优化的 embedding 模型。参考 MTEB 排行榜选择最新模型，如 bge-large、E5 系列、m3e（中文场景）等。

### 2. 搜索索引优化

**2.1 向量存储索引**

![[图片存放/6c5b421a07d915381a6ed76f3763b78a_MD5.png]]

**RAG 管道的关键部分是搜索索引**，它存储了在上一步中获得的向量化内容。

对于 1 万 + 元素规模的高效检索，需要使用近似最近邻（ANN）算法：
- **开源库**：faiss、nmslib、annoy
- **托管方案**：Pinecone、Weaviate、Chroma、OpenSearch、ElasticSearch
- **元数据过滤**：存储日期、来源等元数据，支持条件检索

> [!tip] 想深入了解向量嵌入和数据库？
> 参见 [[vector-embedding-db|向量嵌入与数据库]]，包含：
> - **Embedding 原理**：静态词嵌入→动态上下文嵌入→RAG 时代的新要求
> - **训练任务详解**：MLM 掩码语言模型、NSP 下一句预测、度量学习、对比学习
> - **MTEB 选型指南**：从排行榜开始，关注任务/语言/维度/成本六维度
> - **向量数据库工作机制**：HNSW/IVF/LSH/PQ 索引原理
> - **6 种主流方案对比**：Pinecone/Milvus/Qdrant/Weaviate/Chroma/FAISS
> - **FAISS 实战**：LangChain 索引创建实现细节（from_documents→__from→__add）

**2.2 分层索引**

![[图片存放/21693a67992b27e08d5259cd9433c267_MD5.png]]

在大型数据库的情况下，一个有效的方法是创建两个索引——**摘要索引**和**文档块索引**。检索时先在摘要索引中过滤相关文档，再在子集内搜索具体块。

**2.3 假设性问题与 HyDE**

- **假设性问题**：让 LLM 为每个文档块生成一个假设问题，将问题向量（而非块向量）存入索引。查询时匹配问题向量，检索质量更高（查询与假设问题的语义相似度 > 查询与文档块的相似度）。
- **HyDE（Hypothetical Document Embeddings）**：反向思路，让 LLM 根据查询生成假设回答，将假设回答的向量与查询向量一起使用。

**2.4 内容增强检索**

这两种技术的核心目标都是让发送给 LLM 的上下文更完整、更有用。

- **语句窗口检索器（Sentence Window Retriever）**：每个句子单独嵌入，检索到相关句子后，扩展为前后 k 个句子作为上下文发送到 LLM。

![[图片存放/3c3c93d2bfaa790bb88f0bf4da21047a_MD5.png]]

- **自动合并检索器（父文档检索器，Auto-Merging Retriever）**：文档拆分为小子块，子块与父块有引用关系。检索时获取小子块，如果多个子块链接到同一父块，将父块作为上下文发送给 LLM。

![[图片存放/4476e637f610fd1ebd0eca9e2d05085b_MD5.png]]

> [!compare] 两种检索器对比
> | 维度 | 语句窗口检索器 | 自动合并检索器 |
> |------|---------------|---------------|
> | **最小单元** | 单个句子 | 小子块（多个句子） |
> | **扩展逻辑** | 固定窗口（前后 k 句） | 动态合并（归属同一父块则合并） |
> | **适用场景** | 需要极高检索精度，句子级别匹配 | 文档有自然层级结构，子块可能分散但父块完整 |
> | **优势** | 不会遗漏关键句子 | 保留更完整的语义段落 |
> | **形象比喻** | 用放大镜找到关键句子，然后把周围的"邻居"也叫来帮忙 | 找到几个零散的拼图块，如果发现它们属于同一张大拼图，直接把整张图拿来 |

**2.5 融合检索（混合搜索）**

结合**稀疏检索**（BM25、TF-IDF 等关键词匹配）和**稠密检索**（向量语义搜索），通过**倒数排名融合（Reciprocal Rank Fusion, RRF）**算法合并结果。

![[图片存放/92921a14d1e9207aa22435ed5fca1f39_MD5.png]]

> [!info] 融合检索的优势
> 同时考虑语义相似性和关键词匹配，能捕获两种算法各自擅长但对方可能遗漏的结果。LangChain 的 Ensemble Retriever 和 LlamaIndex 的类似功能都实现了这一策略。

> [!compare] 三种检索器的关系
> | 检索器 | 所属阶段 | 核心目标 |
> |-------|---------|---------|
> | **语句窗口检索器** | 检索后→发送前 | 扩展上下文（单句→窗口） |
> | **自动合并检索器** | 检索后→发送前 | 合并上下文（子块→父块） |
> | **融合检索** | 检索阶段 | 提升召回质量（双算法→RRF 融合） |
>
> **组合使用**：融合检索解决**"找得全"**的问题，后两者解决**"给得准"**的问题。典型管道：融合检索 → 语句窗口/自动合并 → 重排 → 注入 Prompt。

> [!tip] 想了解混合检索的完整实现？
> 参见 [[retrieval-optimization|检索优化技术]]，包含：
> - 稀疏向量 vs 密集向量的详细对比（BM25 公式、OOV 问题）
> - RRF 融合算法的数学原理和代码实现
> - BGE-M3 双向量生成完整示例
> - Milvus 混合检索实战（Schema 定义、索引创建、RRF 参数调优）

### 3. 重排（Reranking）和过滤（Filtering）

检索结果的后处理步骤，在将上下文注入 LLM 之前进一步优化：

- **过滤**：根据相似度阈值、关键词、元数据条件筛除不相关结果
- **重排**：使用交叉编码器（cross-encoder）、Cohere 重排接口、sentence-transformer 等模型对结果重新排序

这是提升最终答案质量的最后一道关卡。

### 4. 查询转换（Query Transformation）

使用 LLM 作为推理引擎修改用户输入，提升检索质量：

![[图片存放/73109aa37a5a92e9ebcc365e5e9703e7_MD5.png]]

| 技术 | 原理 | 适用场景 |
| --- | --- | --- |
| **子查询分解** | 将复杂问题拆分为多个子问题并行检索 | "LangChain 和 LlamaIndex 哪个更受欢迎？"→ 分别查询两者的 Star 数 |
| **Step-back Prompting** | 生成更通用的查询，检索高层次上下文 | 需要背景知识辅助理解的问题 |
| **查询重写** | 重新表述用户查询，改进语义匹配 | 原始查询表述模糊或不规范时 |

> [!tip] 三种技术可以组合使用
> 这三种技术作用于**不同维度**，不仅不互斥，还可以形成强大的组合拳：
>
> ```
> 原始查询
>    ↓
> 【1. Step-back】→ 生成更通用的查询（获取背景知识）
>    ↓
> 【2. 子查询分解】→ 拆分为多个子问题并行检索
>    ↓
> 【3. 查询重写】→ 每个子查询添加关键词、规范化表述
>    ↓
> 并行检索 → 融合 → 重排
> ```
>
> **权衡**：组合使用会显著增加延迟和成本（多次 LLM 调用）。建议：简单查询只用查询重写；复杂查询用子查询 + 重写；高价值场景才三种全用。

> [!tip] 想深入了解查询转换和路由？
> 参见 [[query-rewriting-routing|查询重构与分发]]，包含：
> - **提示工程实战**：让 LLM 生成 JSON 指令解决"最值"查询
> - **多查询分解**：MultiQueryRetriever 并行检索实现
> - **Step-Back Prompting**：Google DeepMind 提出的抽象推理技巧
> - **HyDE**：假设性文档嵌入的代码实现
> - **查询路由**：LLM 意图识别 vs 嵌入相似度路由两种方案

### 5. 聊天引擎（Chat Engine）

支持多轮对话的 RAG 系统，需要处理后续问题、代词指代等场景：

![[图片存放/92d7ce2fa2f23783d56700565cabfc21_MD5.png]]

- **ContextChatEngine**：检索相关上下文 + 聊天记录一起发送给 LLM
- **CondensePlusContextMode**：聊天记录 + 最后一条消息压缩为新查询，检索后将上下文与原始消息一起给 LLM

> [!compare] 两种模式对比
> | 维度 | ContextChatEngine | CondensePlusContextMode |
> |------|------------------|------------------------|
> | **检索用词** | 用户原始最后一句 | LLM 压缩后的新查询 |
> | **LLM 调用次数** | 1 次（生成回答） | 2 次（压缩 + 生成） |
> | **代词处理** | ❌ 可能失效（如"它"、"那个"） | ✅ 自动解析 |
> | **延迟** | 低 | 较高 |
>
> **例子**：用户问"那**它**的吸力有多大？"——ContextChatEngine 用原句检索可能匹配不到"石头 P10"相关内容；CondensePlusContextMode 先压缩为"石头扫地机器人 P10 的吸力参数"再检索，更精准。

> [!tip] 想学习查询构建和元数据过滤？
> 参见 [[query-construction|查询构建]]，包含：
> - **自查询检索器**：SelfQueryRetriever 将自然语言转为结构化过滤器
> - **元数据字段定义**：AttributeInfo 配置最佳实践
> - **文本到 Cypher**：图数据库查询的自然语言转换
> - **复杂查询解决**：LLM 生成 JSON 指令处理排序/比较查询

### 6. 查询路由（Query Routing）

LLM 驱动的决策层，决定"下一步该做什么"：
- 选择使用哪个索引（向量库、图数据库、关系型数据库）
- 选择执行什么操作（总结、搜索、多路分发）

> [!example] 多文档场景
> 定义多个路由选项：总结索引、文档块向量索引、知识库 A、知识库 B。LLM 根据查询语义自动选择最合适的路由。

> [!info] Query Routing vs Agent
> **Query Routing 是 Agent 能力的子集**——两者功能有重叠，但不在一个量级上：
>
> | 维度 | Query Routing | Agent |
> |------|--------------|-------|
> | **核心能力** | 单次决策：选哪个工具/索引 | 多步推理：规划→决策→执行→反思 |
> | **执行步数** | 一步到位（选完就结束） | 可以多步（选 A→结果不够→再选 B） |
> | **工具使用** | 只能选一个（或并行多选） | 可以顺序调用多个工具 |
> | **自我反思** | ❌ 无 | ✅ 可以判断"答案够不够好" |
> | **记忆能力** | ❌ 无状态 | ✅ 可以记住之前的尝试和结果 |
>
> **架构关系**：Query Routing 可作为独立组件（轻量级），也可作为 Agent 的子模块（复杂场景）。

> [!tip] 想学习查询路由的完整实现？
> 参见 [[query-rewriting-routing#二、查询路由|查询重构与分发 — 查询路由]]，包含：
> - **应用场景**：数据源路由、组件路由、提示模板路由
> - **基于 LLM 的意图识别**：RunnableBranch 实现分类 - 分发流水线
> - **嵌入相似性路由**：计算查询与路由描述的余弦相似度
> - **LlamaIndex 实现**：RouterQueryEngine + QueryEngineTool

### 7. 智能体（Agent）

智能体为 LLM 提供推理能力和工具调用能力。在 RAG 场景中：

**多文档智能体方案**——每个文档配一个智能体，具备摘要和问答两种工具；顶层智能体负责路由到各文档智能体，并综合最终答案：

![[图片存放/82d2b177f66271c932f3242d4f2047e5_MD5.png]]

> [!warning] 性能权衡
> 多智能体架构需要多次 LLM 往返调用，运行速度较慢。LLM 调用通常是 RAG 管道中最耗时的操作。对于大型多文档存储，建议简化架构。

### 8. 响应合成（Response Synthesis）

根据所有检索到的上下文生成最终答案的策略：

- **简单拼接**：将所有高于阈值的上下文拼接后一次性给 LLM
- **迭代优化**：逐块发送上下文，让 LLM 逐步完善答案
- **多答案汇总**：根据不同上下文块生成多个答案，再汇总或概括

## RAG 融合（RAG Fusion）

RAG Fusion 是一种提升检索阶段效果的技术：使用 LLM 根据用户查询生成多个相关查询，执行多次向量搜索，然后通过 RRF 算法对结果重新排序。

![[图片存放/31fac67efd32b53c8ebfea9780c36abd_MD5.jpg]]

**优点**：
- 提供多样化的上下文视角
- 自动校正拼写错误、补充隐含信息
- 额外控制层面，可调优应用行为

**缺点**：
- 增加延迟（额外的 LLM 调用）
- 对内部行话/专业术语可能产生误导
- 成本增加（但第一次 LLM 调用成本低，主要是第二次大量 token 的调用）

RAG Fusion 的优缺点对比：

![[图片存放/2f6476705ef89b9be460cb04a703f719_MD5.jpg]]

> [!example] RAG Fusion 不适用场景
> 内容包含大量内部行话时，LLM 可能误解术语含义，生成无关查询。例如"注意力机制"在 Transformer 论文语境下，LLM 可能生成与"专注力""注意力缺陷"相关的无关查询。

![[图片存放/be0d663389c9635da67ce6031c19d312_MD5.jpg]]

由于在这个语境下理解"注意力"的含义依赖于上下文，LLM 对这个含义产生了误解，并生成了一些与之完全无关的搜索查询。

## 微调策略

### 编码器微调

对 embedding 模型（如 bge-large）进行微调以提升检索质量。但实践中，针对搜索优化的最新 Transformer 编码器已经非常高效，微调带来的提升有限。

### 排序器微调

使用交叉编码器对检索结果重排。将查询和每个检索块一起送入交叉编码器，微调使其对相关块输出 1、不相关输出 0。

### LLM 微调

- **OpenAI 微调 API**：Ragas 框架测试显示，微调后的 GPT-3.5-turbo 忠实度指标提升 5%
- **RA-DIT（Retrieval Augmented Dual Instruction Tuning）**：同时微调 LLM 和 Retriever，在知识密集型任务上指标提升约 5%

## 评估体系

RAG 系统评估使用**RAG 三元组**模式：

| 指标 | 关注点 | 优化手段 |
| --- | --- | --- |
| **检索内容相关性** | 检索到的知识是否与问题相关 | 高级 RAG 技术 1-7 + 编码器/排序器微调 |
| **答案忠实度** | 答案是否基于提供的上下文 | 响应合成策略 + LLM 微调 |
| **答案相关性** | 答案是否直接回答问题 | 响应合成策略 + LLM 微调 |

主流评估框架：
- **Ragas**：真实性、答案相关性、上下文精准度/召回率
- **Truelens**：RAG 三元组评估模式
- **LangSmith**：自定义评估器 + 运行监控

## 关键要点

1. **RAG 是解决大模型幻觉和私域知识问题的首选方案**，核心是"检索 + 生成"两阶段架构。
2. **检索质量决定 RAG 上限**——分块策略、索引优化、融合检索、重排等高级技术都服务于提升检索相关性。
3. **高级技术需要权衡**——RAG Fusion、多智能体等复杂架构带来效果提升的同时，也增加延迟和成本，需根据场景选择。
4. **评估先行**——在优化前建立评估体系，用数据驱动决策，避免"提升一组查询效果却牺牲另一组"的陷阱。
5. **速度是生产核心挑战**——流式输出不是噱头而是用户体验刚需，小参数规模 LLM（如 Mixtral、Phi-2）在 RAG 场景有广阔前景。
