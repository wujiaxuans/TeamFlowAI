# CLAUDE.md

减少大型语言模型（LLM）编码常见错误的行為准则。可根据项目特定需求与项目指令合并使用。

**权衡考量：** 这些准则偏向谨慎而非速度。若是极简单的任务，请自行判断。

## 1. 先思考再写代码

**不要假设。不要隐藏困惑。要把权衡点说清楚。**

在实现之前：
- 明确说出你的假设。如果不确定，就直接提问。
- 如果存在多种解读，全部列出来——不要默默选一个。
- 如果有更简单的做法，就说出来。必要时要勇敢推回（push back）。
- 如果有不清楚的地方，就停下来。明确指出哪里困惑，然后提问。

## 2. 简洁优先

**用最少的代码解决问题。不要做任何推测性的功能。**

- 不要新增用户没有要求的额外功能。
- 单次使用的代码不要做抽象化。
- 不要新增未被要求的「弹性」或「可设定性」。
- 不要为不可能发生的情境写错误处理。
- 如果你写了 200 行但其实 50 行就能搞定，就重写。

问问自己：「资深工程师会不会觉得这太复杂？」如果会，就简化。

## 3. 精准修改（Surgical Changes）

**只碰该碰的部分。只清理自己制造的乱七八糟。**

修改现有代码时：
- 不要「顺手优化」旁边的代码、注释或格式。
- 没坏掉的东西不要重构。
- 即使你觉得换个写法更好，也要保持现有风格。
- 如果发现无关的废弃代码，说出来就好——不要自行删除。

当你的修改产生孤儿代码时：
- 只移除因为**你的修改**而变成未使用的 import、变量、函数。
- 除非用户明确要求，否则不要移除原本就存在的废弃代码。

测试标准：每一个被修改的行，都必须能直接追溯到用户的需求。

## 4. 以目标为导向的执行

**先定义成功标准，然后不断验证直到通过。**

把任务转化成可验证的目标：
- 「新增验证」→「先写无效输入的测试，再让测试通过」
- 「修掉这个 bug」→「先写一个能重现 bug 的测试，再让测试通过」
- 「重构 X」→「重构前后测试都要通过」

对于多步骤任务，先简要列出计划：
```
[步骤] → 验证方式：[检查项目]
[步骤] → 验证方式：[检查项目]
[步骤] → 验证方式：[检查项目]
```

强而有力的成功标准能让你独立循环执行。太弱的标准（例如「让它能运作」）则会一直需要澄清。

---

**这些准则运作良好的迹象是：**  
diff 中不必要的变更变少、因为过度复杂而重写的情况减少，以及澄清问题的提问出现在实现之前，而不是在出错之后。
